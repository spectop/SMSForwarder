# CLAUDE.md

本文件定义 SMSForwarder 项目内的开发指导规则，适用于所有参与开发的 agent 与开发者。

## 1. 项目概述

- 项目名称：SMSForwarder
- 技术栈：Kotlin + Android
- 核心流程：短信监听 -> 规则匹配 -> 内容提取 -> 推送
- 配置策略：YAML 配置驱动
- 推送扩展策略：适配器代码静态注册（非动态插件）

## 2. 目标优先级

1. 稳定监听与可靠推送优先于新功能。
2. 规则可配置优先于规则硬编码。
3. 最小可交付优先于过度抽象。

## 3. 必须遵循的工程原则

1. 只做与当前任务直接相关的最小改动。
2. 不得在未说明影响面的情况下修改核心流程。
3. 新增功能必须包含至少一条可验证路径（单测或集成验证）。
4. 配置结构变更必须提供兼容策略或迁移说明。
5. 不得将密钥、Token、账号等敏感信息提交到仓库。

## 4. 目录与文档约定

开发前优先阅读 `.develop` 下文档：

1. `.develop/README.md`
2. `.develop/ARCHITECTURE.md`
3. `.develop/GOALS.md`
4. `.develop/CONSTRAINTS.md`
5. `.develop/WORKFLOW.md`
6. `.develop/PROGRESS.md`

文档更新规则：

- 架构或边界变更：同步更新 `ARCHITECTURE.md`、`CONSTRAINTS.md`
- 里程碑与阶段变化：同步更新 `GOALS.md`
- 任务完成情况：同步更新 `PROGRESS.md`

## 5. 配置与代码约束

1. 配置来源以 YAML 为准，避免把业务规则写死在代码。
2. `PushAdapterRegistry` 采用静态注册，禁止运行时动态加载外部插件。
3. 每个工作流执行需具备超时控制与异常隔离。
4. 网络推送需考虑重试、退避、幂等或去重。

## 6. 安全与隐私

1. 短信正文视为敏感数据，日志默认脱敏。
2. 默认不长期持久化完整短信正文。
3. 对外推送优先使用 HTTPS。
4. 错误日志应可定位问题，但不得泄露密钥。

## 7. 推荐开发节奏

1. 先确认目标与约束，再编码。
2. 先做可验证的最小实现，再扩展。
3. 每个任务结束后更新 `PROGRESS.md` 并记录验证结果。

## 8. 完成定义（DoD）

任务可视为完成需满足：

1. 功能满足需求描述。
2. 不破坏既有流程。
3. 相关测试通过或已说明未覆盖原因。
4. 文档已同步更新。
