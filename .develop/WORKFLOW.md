# 开发流程规范

## 1. 任务生命周期

1. 明确目标：在 `GOALS.md` 中定位任务所属阶段。
2. 明确约束：在 `CONSTRAINTS.md` 中确认边界条件。
3. 实施开发：按最小可交付原则实现功能。
4. 测试验证：先做最小回归，再补充目标场景测试。
5. 更新文档：同步更新 `PROGRESS.md` 与相关架构说明。

## 2. 推荐分支与提交策略

1. 一个任务一个分支，避免混合改动。
2. 提交信息建议使用 Conventional Commits。
3. 每次提交应可独立构建并通过基础测试。

## 3. 变更审查清单

1. 是否引入了与任务无关的重构。
2. 是否破坏现有 YAML 配置兼容性。
3. 是否覆盖异常路径（配置错误、网络失败、超时）。
4. 是否新增或更新了测试。
5. 是否同步更新文档。

## 4. Agent 协作建议

1. 先读 `.develop` 全部文档再修改核心代码。
2. 对高风险变更先给出影响面分析再动手。
3. 对配置结构变更提供迁移说明与样例。
4. 提交前执行最小可用验证并记录结果。
